# 設計コンセプト — GetTennisSponsors / SPONSOR OS

> コンセプト補足ドキュメント。**正本は `sponsor-os-claude-code-handoff.md` と `GOAL_GetTennisSponsors.md`**。本書はその背景・思想を述べる。2026-06-23 確定判断を反映。

---

## 1. 目的とスコープ

テニスのスポンサー獲得を「属人的な営業」から「回り続ける仕組み」へ。**TCがスポンサーシップの仲介業（ブローカー）**になる。

供給側（協賛を募る選手/クラブ/大会）と需要側（協賛企業）を、TC＋エージェントが繋ぎ、成約額から手数料を取る。4つを1システムで扱う：

1. **在庫**：協賛を募集中の対象（選手/クラブ/大会/その他）と連絡先。＝売れるもの。
2. **商談（CRM）**：スポンサー候補／交渉中／契約中の管理。＝デマンド。
3. **実務フロー**：経過報告・成果報告・更新管理まで。＝複利化させる装置。
4. **紹介**：エージェント（TCユーザー）が需要側を連れてくる仕組み。

## 2. 2つの面（サーフェス）

| 面 | 利用者 | できること |
|---|---|---|
| 管理画面（SPONSOR OS） | 佐藤（admin）／staff | 在庫・商談・報告・更新・紹介の全管理 |
| 公開サイト（GetTennisSponsors） | 一般・協賛企業・エージェント | 募集中のティザー閲覧（英語）／相談・申込みフォーム／（Phase 3）紹介リンク付き紹介文のコピー |

「実はTennis Collective運営」の種明かしで信頼とファネルを作る。表側は独立ブランドとして立てる。

**ロール**：`admin`（全データ）／`staff`（担当商談）／`agent`（需要側紹介のみ・報酬対象）。

## 3. 設計思想（なぜこの形か）

- **マッチングは"機能"でなく"ビュー"**。需要はアウトバウンドが主なので、双方向マーケットは需要側が空のまま回らない。提案パイプライン（deals）を背骨にし、マッチングは「企業の課題カテゴリ × sponsorable」のサジェスト層に留める。
- **在庫は一般化**。選手・クラブ・大会は「協賛を売る単位」として本質的に同じ。単一 `sponsorables`（type×sport×jsonb）で持ち、UIだけ分けて見せる。「その他」も jsonb で吸収。
- **品質が商品の信頼を作る**。供給の自己登録は審査ゲートを通す。型が固まる前にエージェントを開放しない（Phase 3を前倒ししない）。
- **複利の装置**。報告（経過→成果）が更新の根拠になり、契約が続くほどTCの取り分が積み上がる。
- **テニス特化で勝つ**。"tennis sponsorship" は競合不在のニッチ。器（sport列）だけ汎用にして、看板はテニスに絞る。

## 4. パイプライン定義

`候補 → アプローチ → 交渉中 → 契約 →（運用中）→ 更新`／離脱は `失注`。
- 運用中：契約後、報告フローが回る区間。
- 更新：`renewal_due` が近づくと自動でこのステージにフラグ。

## 5. 手数料の思想

成約額に対し TCフィー（既定20%）、その中からエージェント配分（5-10%）。TC手残り＝差分。
- **TCは継続して価値を出す**（報告・更新の実務）→ 契約が続く限り毎年取れる（`fee_per_year`）。
- **エージェントの貢献はintroductionに集中** → 初回契約分のみ（`payout_basis=initial_term`）。
- ただし**案件ごとに可変**（強いエージェントや特殊案件は `every_year` に切替可）。

## 7. 多言語・国

- **英語のみ**。UI英語、データ自由テキスト1本、翻訳の並行保持はしない。
- 国は `country`（ISO 3166-1 alpha-2）。表示は英語名、データに国名文字列を直書きしない。
